# PRE_COMPACT_PROMPT.md

Copy-paste prompt này trước khi dùng `/compact`:

```text
Before compacting, create a concise but complete handoff summary for the next assistant.

Purpose:
Preserve only the information needed to continue work safely and accurately after compacting. The summary must be factual, actionable, and security-safe.

Include:

1. Original goal
- What the user ultimately wants to accomplish.
- What project/module this work belongs to.

2. Current status
- What is done.
- What is currently in progress.
- What is blocked.
- What is explicitly not done yet.

3. Validated conclusions only
Include only facts proven by one of:
- reading code/files
- running commands
- tests/smokes
- logs
- DB queries
- browser automation
- user-provided evidence

Do not present guesses as facts. Mark any guess as “unverified”.

4. Important files inspected or changed
For each important file:
- path
- role
- why it matters
- changed/unchanged
- any risk or follow-up

5. Bugs/issues found
For each:
- symptom
- proven or suspected root cause
- evidence
- current status
- whether fixed, worked around, or still open

6. Changes already made
Summarize code/docs/config/test changes already completed.
Do not paste full diffs.

7. Important git checkpoints/commits
For each relevant commit:
- hash
- message
- files included
- purpose
- what was verified before/after commit

8. Current git worktree state
Include:
- output/meaning of latest git status --short
- staged files
- unstaged modified files grouped by task
- untracked files grouped by task
- files that must NOT be accidentally staged/committed
- whether repo is clean

9. Current regression/test phase map
If applicable, list phases and status:
- repo/startup baseline
- infra readiness
- API/auth/session smoke
- UI/static/browser smoke
- realtime/WebRTC path
- STT quality
- LLM provider
- TTS path
- barge-in
- grading/worker
- manual/user validation

For each phase:
- pass/fail/not-run
- key evidence
- remaining risk

10. Remaining steps ordered by priority
For each step:
- goal
- exact next action
- risk
- expected pass condition
- whether it should modify files or only test

11. Constraints / things not to change
Include:
- files/scopes to avoid
- features not to refactor
- commands not to run casually
- whether to avoid stage/commit
- whether to avoid real network/provider calls
- whether to avoid DB/RabbitMQ/Redis writes

12. Operational traps discovered
Include environment/path/API quirks such as:
- commands that must run from a specific working directory
- files that live at repo root vs service dir
- sandbox/network/escalated permission issues
- curl/body format traps
- token/login payload traps
- endpoint/schema quirks
- timeout/finalization timing traps
- dependency/import-time traps
- local artifacts generated by tools
- anything that caused confusion and how to avoid it

13. Exact operational details that materially matter
Include, with secrets redacted:
- commands
- endpoints
- ports
- env vars
- model/provider settings
- queue names
- table names
- service names
- session IDs only if materially needed for debugging
- test fixture paths
- gold transcript paths
- token handling rules

14. Security and secret handling
- Do NOT include real secret values.
- Do NOT include access tokens, JWTs, bearer tokens, API keys, passwords, cookies, full DATABASE_URL values with credentials, login payload secrets, or private credentials.
- If any secret/token was pasted in chat, state that it was exposed and must not be reused or copied into the summary.
- Use placeholders:
  <DATABASE_URL>
  <DB_USER>
  <DB_PASSWORD>
  <JWT_TOKEN>
  <BEARER_TOKEN>
  <API_KEY>
  <COOKIE>
  <SECRET>
  <LOGIN_PAYLOAD>
- Keep real local/dev session IDs only if needed; otherwise use <SESSION_ID>.

15. What is already proven vs what still needs real-world verification
Separate:
- automated/smoke verified
- manually verified
- not verified
- needs user manual testing

16. Recommended next prompt after compact
End with one exact prompt the user can paste immediately after /compact.
The prompt should:
- tell the next assistant what to read/check first
- tell it not to repeat completed cleanup
- include the next safest action
- include “do not modify/stage/commit” if appropriate

Style requirements:
- Keep it concise but complete.
- Keep it factual.
- Do not include unrelated conversation history.
- Do not restate long logs verbatim.
- Do not paste full file contents.
- Do not paste full tokens or login payload contents.
- Use bullet points and short sections.
- Prefer exact paths, commands, and commit hashes when important.
- Clearly mark uncertainty as unverified.
```

 After compact, do not modify, stage, or commit anything until the handoff summary has been reviewed and the current git status is checked.
